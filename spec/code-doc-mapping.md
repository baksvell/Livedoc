# Формат маппинга «код ↔ документация»

Этот документ задаёт, как в документации (Markdown) явно связать фрагмент текста с сущностью в коде, чтобы система могла проверять актуальность и в будущем показывать док в IDE.

## 1. Идентификаторы кода (code_id)

Уникальный идентификатор сущности в коде должен быть однозначно вычисляем парсером и воспроизводим.

### Python (MVP)

Формат: `{module_path}:{qualified_name}` или для простоты MVP: `{file_path}:{name}`.

Примеры:

- `examples.sample_module.calc:add` — функция `add` в модуле `examples.sample_module.calc`
- `examples.sample_module.calc:Calculator.multiply` — метод `multiply` класса `Calculator`

Для MVP достаточно: путь к файлу относительно корня проекта + имя функции/метода (и класс при необходимости). Полный qualified name можно вводить по мере усложнения.

### Расширение на другие языки

- **JavaScript/TypeScript**: `{file_path}:{export_name}` или `{file_path}:{ClassName.methodName}`
- **Go**: `{package}:{FunctionName}` или `{package}:{(*Type).Method}`
- И т.д. — парсер каждого языка выдаёт свой формат code_id, но в графе связей они хранятся как строки.

## 2. Анкоры в документации

Связь «этот параграф/блок описывает эту сущность кода» задаётся анкором в Markdown.

### Синтаксис анкора (HTML-комментарий)

Чтобы не ломать рендер Markdown и не занимать видимый текст, используем **HTML-комментарий** в начале блока (заголовка или параграфа):

```markdown
<!-- livedoc: code_id = "examples.sample_module.calc:add" -->
## Функция add

Складывает два числа...
```

Один анкор может относиться к целому разделу: всё от этого заголовка до следующего заголовка того же или высшего уровня считается «фрагментом документации» для данного `code_id`.

### Несколько сущностей в одном фрагменте

Если параграф описывает несколько сущностей (например, функцию и связанную константу), перечисляем через запятую:

```markdown
<!-- livedoc: code_id = "mymodule:func_a", "mymodule:CONST_X" -->
## API для X

...
```

В MVP достаточно одного `code_id` на блок.

### Идентификатор фрагмента документации (doc_fragment_id)

Для отчётов и графа связей фрагмент доки идентифицируется как:

- `{relative_path_to_md}#{heading_slug}` или
- `{relative_path_to_md}#L{start_line}`

Парсер доков при встрече анкора записывает: `(code_id, doc_fragment_id)`.

## 3. Обратная связь из кода (опционально)

В коде можно указать ссылку на фрагмент документации (для генерации «где описание» и будущей IDE).

### Python: тег в docstring

```python
def add(a: int, b: int) -> int:
    """Складывает a и b.

    .. livedoc:: examples.sample_module.calc:add
    """
    return a + b
```

В MVP можно не реализовывать; достаточно анкоров в Markdown. Обратная ссылка из кода полезна для IDE (показать док по hover).

## 4. Хранение графа связей

- **В памяти**: при каждом запуске `livedoc check` парсеры строят множество пар `(code_id, doc_fragment_id)`.
- **Сохранённое состояние кода**: для каждой `code_id` храним «подпись» (signature hash) — например, хеш нормализованной сигнатуры (имя + аргументы + типы). Файл вида `.livedoc/code_signatures.json` или аналог.
- **Проверка устаревания**: сравниваем текущие подписи с сохранёнными; если для `code_id` подпись изменилась, все связанные `doc_fragment_id` помечаем как «возможно устарело».

## 5. Резюме для MVP

| Элемент | Решение |
|--------|---------|
| Где док | Markdown в `docs/` или рядом с кодом |
| Как связать | HTML-комментарий `<!-- livedoc: code_id = "..." -->` перед заголовком/параграфом |
| code_id (Python) | `{module_path}:{function_or_class.method}` |
| Проверка устаревания | Хеш сигнатуры по коду; при изменении — пометка связанных фрагментов доки |

Этот формат допускает расширение: новые языки добавляют свои правила для `code_id`, парсер доков остаётся общим.
